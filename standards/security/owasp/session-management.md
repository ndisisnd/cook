# Session Management

## NEVER
- Generate session IDs with non-cryptographic PRNGs or with fewer than 64 bits of entropy
- Include sensitive data or PII in session ID values
- Store session state client-side
- Store session tokens in `localStorage` or `sessionStorage` (XSS accessible)
- Use persistent cookies (`Expires`/`Max-Age`) for session management
- Mix HTTP and HTTPS within the same user session
- Reuse session IDs after privilege level change, login, or password change
- Accept session IDs not previously issued by the application
- Rely solely on client-side timeout controls

## ALWAYS
- Generate session IDs with a CSPRNG; minimum 64 bits entropy
- Use opaque, meaningless session IDs; store all state server-side
- Set cookies with `Secure; HttpOnly; SameSite=Strict` (or `Lax`); scope `Domain`/`Path` narrowly
- Enforce HTTPS site-wide; implement HSTS
- Regenerate session ID on every privilege change (auth, password, role, permission)
- Invalidate previous session ID when issuing a new one
- Enforce idle timeout (2–30 min based on risk) and absolute timeout (4–8 h) server-side
- Fully invalidate sessions server-side on logout; clear cookies client-side
- Apply `Cache-Control: no-store` to all responses containing session IDs or sensitive data
- Use framework built-in session management; keep frameworks updated
- Log session events using a salted hash of the session ID (never the raw ID)
- Require reauthentication for: password changes, new/suspicious device, account recovery, privilege escalation

## Cookie Example

```
Set-Cookie: id=<session-id>; Secure; HttpOnly; SameSite=Strict; Path=/app
```

## Session Regeneration (framework calls)

| Framework | Invalidate old | Create new |
| --------- | -------------- | ---------- |
| J2EE | `HttpSession.invalidate()` | `request.getSession(true)` |
| ASP.NET | `Session.Abandon()` | `Response.Cookies.Add(new...)` |
| PHP | `session_destroy()` | `session_start()` + `session_regenerate_id(true)` |

## Attack Detection

- Monitor for session ID brute-force from single IPs
- Detect and alert on session ID anomalies and manipulation attempts
- Optionally bind sessions to IP/User-Agent; alert on mismatch
- Enforce simultaneous session policies per business requirements
- Protect administrative session management interfaces

## Checklist
- [ ] Session IDs generated by CSPRNG with ≥ 64 bits entropy
- [ ] Cookies set with `Secure`, `HttpOnly`, `SameSite`
- [ ] HTTPS enforced site-wide; HSTS header present
- [ ] Session ID regenerated on every privilege change
- [ ] Idle and absolute timeouts enforced server-side
- [ ] Logout fully invalidates server-side session and clears cookie
- [ ] `Cache-Control: no-store` on sensitive responses
- [ ] No session tokens in `localStorage`/`sessionStorage`
- [ ] Session events logged with salted hash of session ID
- [ ] Reauthentication required for high-risk operations
