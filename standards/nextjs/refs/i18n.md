# Internationalization

Use this ref when adding locale routing, translation dictionaries, locale middleware, SEO metadata, `next-intl`, `react-intl`, or `next-translate` patterns.

## Workflow: App Router With `next-intl`

1. Install `next-intl` and create `messages/en.json`, `messages/fr.json`, and other dictionaries.
2. Add locale detection middleware in `middleware.ts`.
3. Create `app/[lang]/layout.tsx` with locale params.
4. Load translations server-side with `getMessages()` or request config.
5. Add `hreflang` tags in `generateMetadata`.
6. Pre-render locales with `generateStaticParams` for static localized pages.
7. Run `next build` and confirm locale paths render.

## Routing And Detection

- Prefer URL-first locale routing for SEO.
- Use dynamic segments such as `app/[lang]/page.tsx` in App Router.
- Use `i18n` config in `next.config.js` for Pages Router.
- Detect language in middleware from `Accept-Language` and cookies, then redirect.
- Use one URL pattern consistently; do not mix sub-paths and domains without intent.

```ts
// middleware.ts
import createMiddleware from 'next-intl/middleware';

export default createMiddleware({
  locales: ['en', 'fr', 'vi'],
  defaultLocale: 'en',
});

export const config = {
  matcher: ['/((?!api|_next|.*\\..*).*)'],
};
```

## Server-Side Messages

Load dictionaries in Server Components or server config to keep client bundles small.

```tsx
// src/i18n.ts
import { getRequestConfig } from 'next-intl/server';

export default getRequestConfig(async ({ locale }) => ({
  messages: (await import(`../messages/${locale}.json`)).default,
}));
```

## Library Selection

- Use `next-intl` for modern App Router projects.
- Use `react-intl` or `next-translate` in legacy apps that already rely on them.

## `next-intl` Principles

- Server-side first: use `getMessages()` and `NextIntlClientProvider` to pass messages to the client.
- Type safety: define message structure for key autocomplete.
- Middleware routing: rely on `next-intl` middleware for automatic locale detection and routing.
- Avoid direct JSON imports in components; use translation hooks or server-loaded messages.
- Avoid client-side manual cookie parsing for locale detection.

## `react-intl` Patterns

- Use `useIntl()` and `intl.formatMessage()` for dynamic strings in functional components.
- Use `<FormattedMessage />` for static JSX text.
- Use hierarchical dot-notated keys such as `cart.order_summary.total`.
- Load only the current locale JSON file for the current request.
- Initialize `IntlProvider` with the server-selected locale and messages to avoid hydration mismatch.

## SEO

- Generate `hreflang` tags through the metadata API for translated routes.
- Ensure canonical URLs match the selected locale pattern.
- Pre-render static localized routes with `generateStaticParams`.

## Anti-Patterns

- Hardcoded user-facing strings in JSX.
- Client-side translation bundles for content that can be loaded server-side.
- Mixed URL locale patterns.
- Inline localized conditionals such as `condition ? 'Yes' : 'No'`; use distinct localized keys.
- Raw strings such as `<span>Submit</span>` in localized UI.
